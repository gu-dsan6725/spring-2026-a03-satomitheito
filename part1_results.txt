Question 4: How does the authentication flow work, from token validation to user authorization?
Category: architecture
Based on the codebase context, I can explain how the authentication flow works from token validation to user authorization. The system uses a multi-layered approach with several authentication providers and fallback mechanisms.

## Authentication Flow Overview

The authentication flow is primarily handled in `./auth_server/server.py` and involves multiple validation strategies:

### 1. Token Extraction and Initial Processing

The flow starts in the `get_current_user_dependency()` function (around line 1151), which:

- Extracts tokens from either `Authorization: Bearer <token>` headers or session cookies
- Supports both `X-Authorization` and `Authorization` headers
- Handles UI sessions (cookies) and CLI/API access (Bearer tokens)

### 2. Multi-Provider Token Validation

The system uses a comprehensive validation approach with multiple fallback methods in `validate_token_comprehensive()` (line 1012):

#### A. Self-Signed Token Validation (Fastest)
- First attempts validation using `validate_self_signed_token()` (line 936)
- Uses internal JWT validation with shared `SECRET_KEY`
- Validates against internal issuer (`JWT_ISSUER`) and audience (`JWT_AUDIENCE`)
- Extracts user groups from JWT claims for OAuth user tokens

#### B. Provider-Specific Validation
The system supports multiple identity providers through the provider pattern:

**Cognito Provider** (`./auth_server/providers/cognito.py`):
- JWT validation using Cognito's JWKS endpoint
- Boto3 fallback for additional validation
- M2M token support with same validation logic

**Entra ID Provider** (`./auth_server/providers/entra.py`):
- JWT validation using Azure AD JWKS
- Self-signed token validation for internal tokens
- Group-based authorization with Azure AD security groups

**Keycloak Provider** (`./auth_server/providers/keycloak.py`):
- Standard OAuth2/OIDC token validation
- Authorization code exchange support

### 3. Static Token Authentication

For Registry API access, the system supports static token authentication:
- Controlled by `REGISTRY_STATIC_TOKEN_AUTH_ENABLED` environment variable
- Uses `REGISTRY_API_TOKEN` for validation
- Bypasses JWT validation when enabled

### 4. Authorization and User Context

After successful token validation, the system:

1. **Extracts User Information**: From validated token claims including:
   - Username
   - Email
   - Groups/roles
   - Token metadata

2. **Creates User Context**: Returns a structured user object with:
   - Authentication status
   - User identity information
   - Group memberships for authorization decisions

3. **Session Management**: 
   - Supports OAuth token storage in session cookies
   - Handles token refresh flows
   - Manages authorization code exchanges

### 5. Authorization Flow Integration

The authentication integrates with OAuth2/OIDC flows:

- **Authorization Code Flow**: Handled in providers for user authentication
- **Device Code Flow**: Supported for CLI applications (Entra ID)
- **Token Exchange**: Authorization codes exchanged for access tokens
- **Scope Management**: Dynamic scope reloading via `/reload-scopes` endpoint

### Key Files and Components

- **Main Auth Logic**: `./auth_server/server.py` - Core authentication and validation
- **Provider Abstraction**: `./auth_server/providers/base.py` - Base provider interface
- **Registry Integration**: `./registry/auth/dependencies.py` - Registry-specific auth dependencies
- **Metrics Auth**: `./metrics-service/app/api/auth.py` - Metrics service authentication

The system is designed to be highly flexible, supporting multiple identity providers while maintaining a consistent authentication interface across all services in the platform.
